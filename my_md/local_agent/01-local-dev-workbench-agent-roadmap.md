# 本地开发工作台 Agent 演进路线

## 1. 背景

当前项目已经具备本地智能 Agent 的核心底座能力：多轮对话、长期记忆、工具调用、插件扩展、Document RAG、工具边界治理、CLI 通信和运行观测。

这些能力使项目不只是普通 RAG demo，而是可以继续演进为面向个人开发者场景的本地开发工作台 Agent：能够理解项目上下文，读取本地文档和源码，调用工具执行任务，并通过运行记录分析每轮行为。

本路线文档用于记录这个演进方向。它不替代 `../rag/` 的 Document RAG 专项文档，也不替代 `../governance/` 的问题治理文档。

## 2. 项目定位

当前项目更准确的定位是：

> 面向本地使用场景的智能 Agent 系统，已经具备本地开发工作台 Agent 的底座能力。

后续如果继续深入“桌面端个人数字员工”方向，建议聚焦在本地开发工作台场景，而不是泛泛地做桌面自动化。这个方向和当前代码、文档问答、工具调用、CLI、运行观测和测试体系最贴近，也更容易形成可验证的工程闭环。

## 3. 当前已具备能力

以下能力来自当前项目已经实现或已在治理文档中验证过的系统能力。

### 多轮对话和上下文组装

系统能够接收用户消息，组装当前 turn 的系统提示、历史上下文、记忆信息和工具信息，并完成模型调用与结果返回。

### 长期记忆和近期上下文管理

系统具备会话历史、长期记忆和近期上下文管理能力，可以在多轮对话中注入必要历史信息，并通过整理和压缩减少无效上下文。

### 工具调用和插件机制

系统支持工具注册、工具调用、工具调用前拦截、生命周期事件监听和插件扩展，使新能力可以在不改动核心对话流程的情况下接入。

### Document RAG

系统已经具备本地文档问答能力，包括文档加载、切分、索引、检索、原文 chunk 展开和 citation 生成。当前工具链以 `search_docs` 和 `fetch_doc_chunk` 为核心。

### 工具访问边界

系统通过 current-turn 工具访问计划限制本轮可见工具、可发现工具和可执行工具，避免强文档问题误转向本地文件读取或命令执行工具。

### TaskPlan 第一阶段

系统已经具备 session 级 active task、步骤持久化、创建/查看/更新工具、active task prompt context、non-LRU 合同，以及 TaskPlan 与后台 job 的 access/completion 边界。真实 smoke 已验证查看和更新计划可在 2 轮内完成，计划创建不再误启动 spawn。

### 工具执行边界

系统通过 turn-local ledger、预算策略和证据充分性判断，控制工具是否还值得真实执行。冗余工具调用可以转为非执行型结果，避免重复执行和证据污染。

### 收尾控制和证据约束

系统能够在文档证据足够时引导模型进入最终回答阶段，并通过证据合同区分原文展开、检索摘要和推断表达，降低回答证据标签失真的风险。

### 运行观测和 trace 查询

系统记录每轮对话的工具链路、模型调用轮次、token 消耗和异常信息，并支持结构化查询历史 turn 的真实工具链。

### CLI 稳定交互

系统已经从简单换行 JSON 通信演进到带长度帧的 CLI 通信协议，并对返回给 CLI 的 metadata 做轻量化处理，降低长工具链导致命令行断连的风险。

## 4. 尚缺能力

以下能力是后续向“本地个人数字员工 / 本地开发工作台 Agent”演进时需要补齐的内容，不应在简历或面试中表述为当前已经完整实现。

### 本地任务规划器

第一阶段已经能够把用户目标保存为持久化步骤，并记录 pending、in-progress、completed、failed、skipped 状态。尚缺的是更完整的暂停、继续、取消、重试、多任务视图、权限审批和任务执行编排，不能把当前状态骨架表述为完整自主任务规划器。

计划创建的上下文召回授权已完成：纯状态创建使用 strict capability scope，不查询 memory；显式偏好或历史各允许一次受限召回。隔离真实 smoke 中纯计划为 `create_task_plan -> final` 2 轮，偏好/历史计划各 3 轮。当前仍缺的是更完整的暂停、取消、重试、多任务视图、权限审批和任务执行编排，而不是上下文授权边界。

### 本地操作权限模型

需要把本地读写、命令执行、网络访问、危险操作和跨目录访问纳入统一权限模型。高风险操作应支持用户确认，默认不能让 Agent 无边界修改本地环境。

### 文件修改确认和回滚

需要在写入文件前生成变更说明或 diff，在删除、覆盖、批量修改前进行确认，并支持备份和回滚。这个能力是桌面端个人数字员工能否安全落地的关键。

### 开发任务闭环

需要形成更完整的本地开发工作流，例如读取报错、查日志、定位源码、生成修复计划、修改文件、运行测试、总结结果。当前项目已有底座，但还需要把这些步骤组合成稳定的任务闭环。

### 外部应用集成

后续可以选择性接入浏览器、IDE、笔记、日历或本地文件索引。但第一阶段不应追求泛化跨应用控制，应先把开发工作台场景做深。

### 持续任务和主动能力

后续可以加入定时检查、项目状态巡检、文件变化监听、每日总结和任务完成通知等能力，使 Agent 从被动问答扩展到可控的主动辅助。

### 更明确的用户工作流

需要围绕一个具体场景打磨端到端体验。推荐优先选择“本地开发工作台 Agent”，而不是同时覆盖文件整理、浏览器代办、日历管理等多个方向。

## 5. 推荐建设优先级

P1：任务规划器第二阶段。
第一阶段状态骨架已完成；下一步先完善上下文需求授权、任务恢复和执行编排，再扩展暂停、取消、重试与多任务视图。

P2：本地操作权限模型。
先定义哪些目录、命令、工具和操作需要允许、拒绝或用户确认。

P3：文件修改确认与回滚。
让 Agent 修改文件前必须形成可审查变更，并能恢复。

P4：开发任务闭环。
打通“读日志 -> 查源码 -> 生成计划 -> 修改文件 -> 运行测试 -> 总结结果”的最小闭环。

P5：外部应用集成。
在本地开发任务闭环稳定后，再考虑浏览器、IDE 或其他桌面应用接入。

P6：主动任务和持续任务。
在权限模型和回滚机制稳定后，再增加定时巡检、文件变化监听和主动通知。

## 6. 与现有模块的关系

Document RAG 是本地知识检索和证据层。它负责从项目文档中找依据，并提供可引用来源。

Tool Access Gateway 是工具访问边界。它负责当前 turn 哪些工具能出现、能被发现、能执行。

Turn Tool Boundary 是工具执行边界。它负责判断一个工具调用是否还值得真实执行。

Turn Completion 是收尾控制。它负责判断是否应该停止 ReAct 工具循环，进入最终回答阶段。

Evidence Contract 是回答证据约束。它负责限制最终回答中哪些内容能称为原文、哪些只能称为检索摘要或推断。

Observe / Trace 是运行可观测性。它负责记录工具链路、模型调用轮次、token 消耗和异常信息，并为问题排查提供事实来源。

CLI IPC 是本地交互通道。它负责在本地命令行和 Agent 服务之间稳定传递用户输入、模型输出和轻量 metadata。

## 7. 简历表达方式

推荐把项目描述为：

> 面向本地开发场景设计并实现智能 Agent 系统，支持多轮对话、长期记忆、工具调用、插件扩展、本地文档问答、命令行交互和运行轨迹记录。项目重点解决本地 Agent 在多工具环境下的上下文管理、工具调用边界、文档证据链和运行稳定性问题，具备向本地开发工作台 Agent 演进的底座能力。

表述时应注意：

- 可以说“具备本地开发工作台 Agent 的底座能力”。
- 不应说“已经实现完整桌面端个人数字员工”。
- 可以强调工具边界治理、证据充分性、运行观测和 CLI 稳定性。
- 不应把尚未实现的浏览器操作、IDE 图形界面控制、文件回滚闭环写成已完成能力。

## 8. 面试可讲技术点

为什么不是普通 RAG demo：
项目不仅能检索文档，还治理了工具可见性、工具执行边界、证据充分性、提前收尾、运行观测和 CLI 稳定性。

为什么工具调用需要边界治理：
模型在复杂任务中容易重复调用、错误调用或转向不合适工具。系统需要在当前 turn 控制工具可见性、发现路径和执行权限。

本地 Agent 为什么需要权限模型：
桌面端个人数字员工会接触本地文件、命令和外部应用。如果没有权限模型和确认机制，Agent 很容易产生安全风险或不可恢复的本地副作用。

如何处理工具链过长：
通过工具访问边界、执行预算、证据充分性判断和最终回答收尾控制，把固定路径任务从开放式探索收束为可控工具链。

如何定位 Agent 行为问题：
通过 observe 记录、工具链 trace、模型调用轮次、token 统计和结构化 turn trace 查询，定位某一轮为什么调用了某个工具、是否被边界拦截、是否进入最终回答。

如何从当前系统演进到个人数字员工：
先补任务规划、权限模型、文件修改确认和开发任务闭环，再考虑浏览器、IDE 和其他桌面应用集成。

## 9. 非目标

当前项目不声称已经完整实现以下能力：

- 完整桌面自动化。
- 浏览器自主操作和表单提交。
- IDE 图形界面控制。
- 文件修改确认和回滚闭环。
- 完整跨应用状态同步。
- 多应用长期后台任务编排。

这些能力可以作为后续路线，但不应混入当前已实现能力描述。
